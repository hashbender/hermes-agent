# 接入方式功能差异对比

> **面向**：技术 + 产品
> **目的**：把 `api_server`（当前公司采用的接入方式）与 hermes 的其他接入方式（CLI、各消息平台、ACP 编辑器集成、Cron）放在一起做能力对比，回答两个问题：
> 1. **当前接入方式缺什么**？（AskUserQuestion 只是其一，还有哪些？）
> 2. **当前接入方式独有什么**？（有没有别的接入方式做不到的能力？）
>
> **工具逐项清单会随版本变化**，本文以「能力类别」对比为主；需要逐工具核对时，运行 `hermes chat --list-toolsets` / `hermes chat --list-tools` 实测。

---

## 1. 接入方式总览

hermes 通过 `platform` 概念支持多种接入，每种绑定一个「工具集预设」（源码：`hermes_cli/platforms.py`、`toolsets.py`）：

| 接入方式 | 工具集预设 | 定位 | 谁在用 |
|----------|-----------|------|--------|
| **`api_server`** | `hermes-api-server` | OpenAI 兼容 HTTP 服务端 | **公司通用层（当前接入）** |
| CLI | `hermes-cli` | 终端交互式 TUI | 开发者本地 |
| ACP | `hermes-acp` | 编辑器集成（VS Code/Zed/JetBrains） | 编辑器插件 |
| Telegram / Discord / WhatsApp / Slack / Signal / Weixin / QQBot / Yuanbao | 各自预设 | 消息平台 Bot | IM 用户 |
| Webhook | `hermes-webhook` | 通用 webhook | 外部系统集成 |
| Cron | `hermes-cron` | 定时任务 | 无人值守调度 |

> **消息平台与 CLI 共享同一套核心工具 `_HERMES_CORE_TOOLS`（`toolsets.py:31`），其中含 `clarify`（`toolsets.py:33`）。`api_server` 与 `acp` 是裁剪版。**

## 3. 当前接入方式（`api_server`）「不能做」清单

### 3.1 AskUserQuestion（`clarify`）—— 核心痛点

详见 [文档 01](./01-不支持AskUserQuestion的原因.md)。一句话：工具集白名单裁掉 + 未注入回调 + 无异步通道，agent 无法在执行中途向用户提问。

### 3.2 主动推送消息

`api_server` 是**请求-响应**模型：客户端问、服务端答。agent **无法主动**向用户发消息（没有消息平台的推送通道）。

> 注：所有接入方式的 agent 都没有 `send_message` 这个「工具」（源码注释明确：主动推送由平台层处理，不在 agent 循环内）。但消息平台天然有推送通道，`api_server` 没有——所以消息平台能做的「agent 完成后主动通知」「定时推送」等，`api_server` 做不到，需客户端轮询或长连接。

### 3.3 原生 UI 渲染（按钮 / 卡片）

`api_server` 只输出**数据**（JSON / SSE 事件），不输出**渲染好的 UI**。按钮、卡片、内联键盘都得**客户端自己画**。
- 消息平台：hermes 直接渲染 Telegram 内联键盘、Discord 按钮、WhatsApp 按钮/列表；
- `api_server`：客户端收到 `tool.started` 等事件后，自行决定怎么展示。

> **这对产品需求的启示**：AskUserQuestion 卡片即便能力补齐，hermes 也只负责「发事件」，卡片渲染仍由业务层/通用层负责——这本来就是你们要做的工作。

### 3.4 语音输入 / Voice Mode

语音模式（录音、TTS 播报、语音键）是 CLI/GUI 独有的交互形态，`api_server` 不提供（音频流不在 OpenAI 兼容协议范围内）。

### 3.5 平台插件工具

`discord_admin`（管 Discord 服务器）、`feishu_doc_read`（读飞书文档）、`spotify`（控制播放）、`homeassistant`（智能家居）、`yuanbao_*` 等，是各平台专属工具，`api_server` 默认不暴露。

## 4. 当前接入方式（`api_server`）「独有」清单

这部分常被忽略，但对技术选型很关键——`api_server` 不是「阉割版」，而是「**为程序化集成优化的接入方式**」，有别的接入方式都没有的能力：

### 4.1 OpenAI 兼容标准协议

`/v1/chat/completions`、`/v1/responses` 完全兼容 OpenAI 规范。**任何支持 OpenAI 的客户端**（NextChat、ChatBox、AnythingLLM、OpenAI SDK 各语言版）都能**零改动**接入（源码：`api_server.py:26`）。这是公司「通用层」选择它的根本原因——标准化、语言无关、生态复用。

### 4.2 `/v1/runs` 结构化事件流（最丰富）

`/v1/chat/completions` 的 SSE 只有 token 流；`/v1/runs` 提供**完整生命周期事件**（源码：`api_server.py:3878-4141`）：

```
run.started → message.started → message.delta… → tool.started → tool.completed
            → … → assistant.completed → run.completed
```

客户端能精确知道「agent 正在调哪个工具、参数是什么、结果如何、是否在审批」。这是 CLI/消息平台都没有的「程序可消费的结构化事件」。

### 4.3 Responses API（带状态续接）

`POST /v1/responses` 支持 `previous_response_id` 续接历史，hermes 侧用 SQLite `ResponseStore` 持久化（`api_server.py:372`）。客户端不必自己维护完整 message 历史，只传一个 id 即可。

### 4.4 `approval` 异步通道（唯一的「中途暂停」能力）

这是 `api_server` 下**唯一**实现了「暂停 agent → 等用户 → 继续」语义的能力（源码：`api_server.py:4030-4088`）：

- SSE 推 `approval.request` 事件；
- 客户端 `POST /v1/runs/{run_id}/approval` 回填决策；
- agent 唤醒继续。

**这正是 AskUserQuestion 应当复刻的范式**（见文档 01 第 5 节）。

### 4.5 生产级工程能力

- **并发控制**：`max_concurrent_runs` 限流（`api_server.py` `_concurrency_limited_response`）；
- **幂等性**：`Idempotency-Key` 头防重复（`api_server.py:634`）；
- **能力自描述**：`GET /v1/capabilities` 机器可读（`api_server.py:1213`）；
- **统一鉴权**：Bearer token，所有端点统一 `_check_auth`；
- **会话隔离**：`X-Hermes-Session-Id` / `X-Hermes-Session-Key` 头。

这些都是 CLI/消息平台不提供的「服务端治理」能力。

## 5. AskUserQuestion 在各接入方式的落点

| 接入方式 | AskUserQuestion | 渲染形式 | 机制 |
|----------|:--------------:|---------|------|
| CLI | ✅ | TUI 选择面板（↑↓ + 数字键 + Other 输入） | `_clarify_callback` + prompt_toolkit |
| Telegram | ✅ | 内联键盘按钮 + Other | `send_clarify` + 按钮回调 |
| Discord | ✅ | 按钮 View + Other | `send_clarify` + `ClarifyChoiceView` |
| WhatsApp | ✅ | 按钮 / 列表消息 | `send_clarify` 按钮或列表 |
| Slack / Signal / 其他 | ✅ | 编号文本列表（base 默认实现） | text-fallback + 文本拦截 |
| **`api_server`** | ❌ | — | **无工具、无回调、无通道** |
| ACP 编辑器 | ❌ | — | 工具集裁剪 |
| Cron | ⚠️ | 无人值守，返回哨兵自决 | `_oneshot_clarify_callback` |

> 注：Cron / 无人值守场景下 clarify 即使被调，也会返回「无用户可用，请自行决定」的哨兵（`oneshot.py:376`），不会真提问——这是合理的，因为那些场景本就没有交互对象。

## 6. 其他接入方式独有能力速览

| 接入方式 | 独有能力 |
|----------|---------|
| **CLI / GUI** | 完整 TUI（面板、键盘导航、`/` 命令、model picker、sudo 提词）、Voice Mode、cronjob 管理、读终端标签页 |
| **ACP 编辑器** | 编辑器深度集成、diff 预览 / apply 工作流、项目上下文感知 |

## 7. 落地路径决策建议

针对「新增 AskUserQuestion 卡片」需求，结合本对比：

| 路径 | 做法 | 可靠性 | 代价 | 适用场景 |
|------|------|:------:|------|----------|
| **A. hermes 侧补齐 clarify（正道）** | 仿 `approval` 给 `clarify` 接「SSE 事件 + POST 回填 + notify + 工具集加回 + 注入回调」五件套 | 高（工具级） | 需 hermes 权限/排期 | 长期方案，能力完整 |
| **B. 通用层模拟协议** | system prompt 约束模型用结构化文本表达疑问，通用层在 SSE 流里拦截、渲染卡片、下一轮带入答案 | 中（依赖模型守约） | 通用层中等工作量 | 短期落地，不依赖 hermes 改造 |
| **C. 业务层前置表单** | 业务流程先弹表单收集关键决策，拼进 prompt 再调 | 高 | 最低，但语义弱化 | 决策点少且固定的场景 |

> **推荐**：短期走 B 或 C 快速上线 MVP；同步推动 A 作为长期正道。文档 02 的本地 demo 可作为向产品/技术演示「当前边界」与「补齐后效果」的对照环境（补齐 clarify 后，用同一套 demo 即可看到 `clarify.request` 事件）。
