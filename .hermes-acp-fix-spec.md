# ACP v1 Comprehensive Fix Specification

## Context

This is an implementation specification for fixing all gaps between Hermes Agent's ACP implementation and the official ACP v1 specification (https://agentclientprotocol.com).

### Codebase
- Hermes Agent repo at `~/.hermes/hermes-agent/`
- ACP adapter lives in `acp_adapter/server.py` (~2296 lines)
- ACP entry point: `acp_adapter/entry.py`
- ACP tests: `tests/acp/`

### What's Already Done (committed to branch `eva/acp-comprehensive-fix`)
- `config_options` in session responses (model selector for Zed)
- `--skills` flag support in ACP mode
- MCP toolsets preserved on model switch (`set_session_model` + `_cmd_model`)
- `/skill` slash command (list/load)
- Test updates for config_options and advertised commands

### Python ACP Library
Version: `agent-client-protocol==0.9.0` — supports the full ACP v1 schema with types for all protocol messages.

## Task: Implement ALL remaining ACP v1 gaps

Please fix the following issues in ORDER OF PRIORITY:

### PRIORITY 1 — Critical (User-facing bugs)

#### 1a. Fix `_extract_text()` to handle ALL content block types

**File:** `acp_adapter/server.py`, function `_extract_text()` (line ~373)
**Problem:** Only handles `TextContentBlock`. Silently drops `ResourceContentBlock`, `ImageContentBlock`, `AudioContentBlock`, and `EmbeddedResourceContentBlock`. This is the fallback path when content block conversion produces empty parts — so when a user ONLY @-references files (tagged files) without typing text, the entire prompt becomes empty.
**Fix:** Use `_content_blocks_to_openai_user_content()` instead, or update `_extract_text()` to handle all block types with proper resource-to-text conversion.

#### 1b. Handle ACP `ContentBlock` union from prompt correctly

**File:** `acp_adapter/server.py`, function `prompt()` (line ~1400+)
**Problem:** The `prompt()` method receives content blocks from the client but may not convert ALL block types before passing to the agent. Verify `_content_blocks_to_openai_user_content` is called properly and resources are always included.
**Verification:** When Zed sends an @-referenced file as a `ResourceContentBlock` with `file://` URI, the agent must see the file content in its context.

---

### PRIORITY 2 — High (Missing spec features)

#### 2a. Send plan entry status updates

**File:** `acp_adapter/server.py`
**Problem:** Agent plan entries are created (via `_build_plan_update_from_todo_result`) but their `status` property is never updated from `pending` → `in_progress` → `completed`. The ACP spec requires status progression.
**Fix:** After creating plan entries, send `session/update` with updated `status` as each task progresses.

#### 2b. Initialize response: complete agent capabilities

**File:** `acp_adapter/server.py`, method `initialize()` (line ~947)
**Problem:**
- `promptCapabilities` is `image=True` but missing `audio=False` and `embeddedContext=True` (resources ARE supported)
- `agentCapabilities` uses full `PromptCapabilities` but doesn't include terminal capabilities
- Response format should match spec: `protocolVersion`, `agentCapabilities`, `agentInfo`, `authMethods`

**Fix:** Add `audio=False, embeddedContext=True` to promptCapabilities.

#### 2c. Config options: modes as configOptions with category="mode"

**File:** `acp_adapter/server.py`, method `_build_config_options()`
**Problem:** ACP v1 spec says modes SHOULD also be exposed via `configOptions` with `category="mode"`. Currently only model selector is exposed as configOptions.
**Fix:** Add mode selection as additional config options entries with `category="mode"`.

---

### PRIORITY 3 — Medium (Spec compliance)

#### 3a. Advertised commands: add displayName and icon

**File:** `acp_adapter/server.py`, `_ADVERTISED_COMMANDS`
**Problem:** The ACP spec says each command has: `name`, `displayName`, `description`, `toolTip`, `icon`, `arguments`. Currently only `name` and `description` (and sometimes `input_hint`) are provided.
**Fix:** Add `displayName` (human-readable), `toolTip` (optional detail), and appropriate `icon` fields.

#### 3b. Initialize: mcpCapabilities with stdio transport

**File:** `acp_adapter/server.py`, method `initialize()`
**Problem:** Agent capabilities should advertise `mcpCapabilities` (http, sse) per MCP specification. Hermes uses stdio for MCP internally.
**Fix:** Not critical — Hermes manages MCP internally. Skip unless the ACP client needs it.

#### 3c. Environment variable passthrough for MCP servers

**File:** `acp_adapter/server.py`, method `_register_session_mcp_servers()`
**Problem:** When Zed passes `mcpServers` with `env` field (array of key-value pairs), Hermes should set these as environment variables for the MCP subprocess.
**Fix:** Parse the `env` field and merge into the subprocess environment.

### PRIORITY 4 — Low (Edge cases)

#### 4a. Handle empty prompt gracefully

When prompt has zero content blocks, or all blocks fail conversion, return `PromptResponse(stop_reason="end_turn")` with a descriptive error message instead of crashing.

#### 4b. Test coverage for resource content blocks

Add unit tests for:
- ResourceContentBlock with `file://` URI → text conversion
- EmbeddedResourceContentBlock with TextResourceContents
- EmbeddedResourceContentBlock with BlobResourceContents
- Mixed content blocks (text + resource + image)
- Empty/failed resource reads
